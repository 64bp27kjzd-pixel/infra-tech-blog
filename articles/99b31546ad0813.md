## 1. はじめに

前回の記事では、AWS Well-Architectedフレームワークの6つの柱で構成を自己評価し、「動く構成」と「運用に耐える構成」の間にある具体的なギャップを整理しました。

その中で、**優先度が高い改善項目として以下の4点**を挙げました。

1. **CloudWatch監視・アラート**→ 障害に気づけない状態を解消
2. **Secrets Managerによる認証情報管理**→ tfvarsでの平文管理リスクを排除
3. **ASGスケーリングポリシーの定義**→ ASGを「使える」状態にする
4. **HTTPS対応**（Route53 + ACM）→ ドメイン取得費用が発生するため後続で対応

今回は**①のCloudWatch監視・アラート**を実装します。

具体的には以下の3つを追加します。

- **SNSによるメール通知基盤**の構築
- **CloudWatchアラーム**（EC2 / ALB / RDS / NAT Gateway）の追加
- **CloudWatch Logsへのログ集約**（EC2アプリログ・VPCフローログ）

> 「障害が起きたことに気づけない」状態を解消することが、この記事の目的です。

## 2. 追加するモジュール構成

今回新たに `sns` / `logs` / `alarm` の3モジュールを追加します。

```
modules/
├── vpc/
├── ec2/
├── alb/
├── rds/
├── sns/       ← 新規追加
├── logs/      ← 新規追加
└── alarm/     ← 新規追加
```

モジュール間のデータフローは以下の通りです。

| 渡し元 | output | 渡し先 | 用途 |
|---|---|---|---|
| sns | `sns_topic_arn` | alarm | アラーム発報先のSNS Topic |
| vpc | `vpc_id` | logs | VPCフローログの収集対象 |
| vpc | `nat_gw_ids` | alarm | NAT GatewayのCWアラーム対象 |
| ec2 | `asg_name` | alarm | EC2 CWアラームのディメンション |
| alb | `alb_arn` | alarm | ALB CWアラームのディメンション |
| alb | `alb_tg_arns` | alarm | TargetGroupのCWアラームのディメンション |
| rds | `db_id` | alarm | RDS CWアラームのディメンション |

既存モジュールへの変更を最小限に抑えながら、監視基盤をモジュールとして追加する構成です。

## 3. SNSモジュール

まず、アラーム発報時の通知先となるSNS Topicとメールサブスクリプションを作成します。

```terraform
# SNS Topicの作成
resource "aws_sns_topic" "send_to_email_topic" {
  name = "${var.env}-send_to_email_topic"
}

# SNS Subscription
resource "aws_sns_topic_subscription" "send_to_email_subscription" {
  topic_arn = aws_sns_topic.send_to_email_topic.arn
  protocol  = "email"
  endpoint  = var.send_email
}
```

### ポイント

SNS（Simple Notification Service）は、CloudWatchアラームの通知先として定番の構成です。

`protocol = "email"` を指定することで、アラーム発報時に指定したメールアドレスへ通知が届きます。

**初回 `terraform apply` 後に確認メールが届くため、メール内のリンクから手動で承認（Subscribe）が必要**です。承認前はSNSからの通知が届かないため注意してください。

後続のアラームモジュールは `sns_topic_arn` を受け取り、各アラームの `alarm_actions` に指定します。これにより、すべてのアラームが同一の通知先へ集約されます。

## 4. アラームモジュール

CloudWatchアラームは、EC2・ALB・RDS・NAT Gatewayの4対象に分けて実装しています。

### 4-1. EC2アラーム

```terraform
# EC2 CPU使用率監視
resource "aws_cloudwatch_metric_alarm" "ec2_cpu_alarm" {
  alarm_name          = "${var.env}-asg-${var.asg_name}-cpu-high"
  comparison_operator = "GreaterThanOrEqualToThreshold"
  evaluation_periods  = 2
  metric_name         = "CPUUtilization"
  namespace           = "AWS/EC2"
  period              = 300
  statistic           = "Average"
  threshold           = 70
  alarm_description   = "EC2 CPU使用率が70%を超過 (${var.asg_name})"
  alarm_actions       = [var.sns_topic_arn]

  dimensions = {
    AutoScalingGroupName = var.asg_name
  }
}

# EC2死活監視
resource "aws_cloudwatch_metric_alarm" "ec2_StatusCheck_alarm" {
  alarm_name          = "${var.env}-ec2-${var.asg_name}-status-check-failed"
  metric_name         = "StatusCheckFailed"
  comparison_operator = "GreaterThanOrEqualToThreshold"
  evaluation_periods  = 2
  namespace           = "AWS/EC2"
  statistic           = "Maximum"
  period              = 300
  threshold           = 1
  alarm_description   = "EC2 ステータスチェック失敗 (${var.asg_name})"
  alarm_actions       = [var.sns_topic_arn]
  treat_missing_data  = "missing"

  dimensions = {
    AutoScalingGroupName = var.asg_name
  }
}
```

**CPU使用率アラーム**は、5分間の平均値が70%以上の状態を2回連続で検知したタイミングで発報します。ASGディメンションを指定することで、ASG内の全インスタンスを対象にした集約メトリクスを監視します。

**死活監視アラーム**は `StatusCheckFailed` メトリクスを使用します。EC2のシステムステータスチェック（AWSの基盤側）とインスタンスステータスチェック（OS側）の両方をまとめて監視できます。`statistic = "Maximum"` を指定することで、ASG内のいずれかのインスタンスで1件でも失敗が発生した場合に検知できます。

`treat_missing_data = "missing"` は、インスタンスが存在しない期間（スケールイン後など）にアラームが誤発報しないよう設定しています。

### 4-2. ALBアラーム

```terraform
# ALB UnHealthyHostCount
resource "aws_cloudwatch_metric_alarm" "alb_unhealthy_host" {
  alarm_name          = "${var.env}-alb-unhealthy-host"
  comparison_operator = "GreaterThanOrEqualToThreshold"
  evaluation_periods  = 1
  threshold           = 1
  namespace           = "AWS/ApplicationELB"
  metric_name         = "UnHealthyHostCount"
  statistic           = "Average"
  period              = 60
  dimensions = {
    LoadBalancer = var.alb_arn
    TargetGroup  = var.alb_tg_arn
  }
  alarm_description = "Unhealthy host detected"
  alarm_actions     = [var.sns_topic_arn]
}

# ALB HTTPCode_Target_5XX_Count
resource "aws_cloudwatch_metric_alarm" "alb_target_5xx" {
  alarm_name          = "${var.env}-alb-5xx-error"
  comparison_operator = "GreaterThanOrEqualToThreshold"
  evaluation_periods  = 1
  threshold           = 5
  namespace           = "AWS/ApplicationELB"
  metric_name         = "HTTPCode_Target_5XX_Count"
  statistic           = "Sum"
  period              = 60
  dimensions = {
    LoadBalancer = var.alb_arn
    TargetGroup  = var.alb_tg_arn
  }
  alarm_description = "Too many target 5XX errors"
  alarm_actions     = [var.sns_topic_arn]
}

# ALB TargetResponseTime
resource "aws_cloudwatch_metric_alarm" "alb_response_time" {
  alarm_name          = "${var.env}-alb-response-time-high"
  comparison_operator = "GreaterThanOrEqualToThreshold"
  evaluation_periods  = 2
  threshold           = 1
  namespace           = "AWS/ApplicationELB"
  metric_name         = "TargetResponseTime"
  statistic           = "Average"
  period              = 60
  dimensions = {
    LoadBalancer = var.alb_arn
  }
  alarm_description = "ALB target response time too high"
  alarm_actions     = [var.sns_topic_arn]
}
```

ALBでは3種類のメトリクスを監視しています。

**UnHealthyHostCount**は、ヘルスチェックに失敗しているEC2インスタンスの数を監視します。1台でも異常になった時点で即座に検知したいため、`evaluation_periods = 1` / `period = 60` と短い間隔で設定しています。

**HTTPCode_Target_5XX_Count**は、バックエンドのEC2が返した5xxエラーの件数を監視します。1分間に5件以上の5xxエラーが発生した場合に発報します。ALB自身が返す4xxや5xx（`HTTPCode_ELB_5XX_Count`）とは別のメトリクスで、アプリケーション側の異常を検知するために使用しています。

**TargetResponseTime**は、ALBからEC2への応答時間（秒）の平均値を監視します。1秒を超えた状態が2分続いた場合に発報します。CPU異常やDBの詰まりを間接的に検知できるメトリクスです。

### 4-3. RDSアラーム

```terraform
# RDS CPU
resource "aws_cloudwatch_metric_alarm" "cpu_utilization_too_high" {
  alarm_name          = "${var.env}-rds-cpu-high"
  comparison_operator = "GreaterThanThreshold"
  evaluation_periods  = 1
  metric_name         = "CPUUtilization"
  namespace           = "AWS/RDS"
  period              = 600
  statistic           = "Average"
  threshold           = var.cpu_utilization_threshold
  alarm_description   = "Average database CPU utilization over last 10 minutes"
  alarm_actions       = [var.sns_topic_arn]
  dimensions = {
    DBInstanceIdentifier = var.db_instance_id
  }
}

# RDS 接続数
resource "aws_cloudwatch_metric_alarm" "rds_connections" {
  alarm_name          = "${var.env}-rds-connections-high"
  alarm_description   = "RDS connection count too high"
  namespace           = "AWS/RDS"
  metric_name         = "DatabaseConnections"
  comparison_operator = "GreaterThanOrEqualToThreshold"
  threshold           = var.db_connections_threshold
  evaluation_periods  = 2
  period              = 60
  statistic           = "Average"
  dimensions = {
    DBInstanceIdentifier = var.db_instance_id
  }
  alarm_actions      = [var.sns_topic_arn]
  treat_missing_data = "notBreaching"
}

# RDS ストレージ残量
resource "aws_cloudwatch_metric_alarm" "free_storage_space_too_low" {
  alarm_name          = "${var.env}-rds-storage-low"
  comparison_operator = "LessThanThreshold"
  evaluation_periods  = 1
  metric_name         = "FreeStorageSpace"
  namespace           = "AWS/RDS"
  period              = 600
  statistic           = "Average"
  threshold           = var.free_storage_space_threshold
  alarm_description   = "Average database free storage space over last 10 minutes"
  alarm_actions       = [var.sns_topic_arn]
  dimensions = {
    DBInstanceIdentifier = var.db_instance_id
  }
}

# RDS メモリ残量
resource "aws_cloudwatch_metric_alarm" "freeable_memory_too_low" {
  alarm_name          = "${var.env}-rds-memory-low"
  comparison_operator = "LessThanThreshold"
  evaluation_periods  = 1
  metric_name         = "FreeableMemory"
  namespace           = "AWS/RDS"
  period              = 600
  statistic           = "Average"
  threshold           = var.freeable_memory_threshold
  alarm_description   = "Average database freeable memory over last 10 minutes too low"
  alarm_actions       = [var.sns_topic_arn]
  dimensions = {
    DBInstanceIdentifier = var.db_instance_id
  }
}
```

RDSでは4つのメトリクスを監視しています。CPU・接続数・ストレージ残量・メモリ残量です。

**CPU**と**メモリ・ストレージ残量**はいずれも `period = 600`（10分間隔）で集計しています。DBのリソース枯渇は緩やかに進行することが多いため、短い間隔でのポーリングよりも10分集計の方がノイズを抑えた検知ができます。

**接続数**は `period = 60` で短めに設定しています。接続プールの枯渇はアプリケーション全体に即座に影響するため、早期検知を優先しています。`treat_missing_data = "notBreaching"` は、RDSが停止中など接続数メトリクスが取得できない期間に誤発報しないための設定です。

CPUの閾値（`var.cpu_utilization_threshold`）、接続数の閾値（`var.db_connections_threshold`）、ストレージ・メモリの閾値はすべてvariablesで受け取る設計としており、dev / prod で異なる値を設定できます。

### 4-4. NAT Gatewayアラーム

```terraform
# NAT Gateway ポート不足
resource "aws_cloudwatch_metric_alarm" "natgw_error_port_allocation" {
  for_each            = var.nat_gw_ids
  alarm_name          = "${var.env}-natgw-${each.value}-port-allocation-error"
  alarm_description   = "NAT Gateway port allocation errors detected"
  namespace           = "AWS/NATGateway"
  metric_name         = "ErrorPortAllocation"
  comparison_operator = "GreaterThanOrEqualToThreshold"
  threshold           = 1
  evaluation_periods  = 1
  period              = 60
  statistic           = "Sum"
  dimensions = {
    NatGatewayId = each.value
  }
  alarm_actions = [var.sns_topic_arn]
}

# NAT Gateway ドロップしたパケット数
resource "aws_cloudwatch_metric_alarm" "natgw_packet_drop" {
  for_each            = var.nat_gw_ids
  alarm_name          = "${var.env}-natgw-${each.value}-packet-drop"
  alarm_description   = "NAT Gateway packet drops detected"
  namespace           = "AWS/NATGateway"
  metric_name         = "PacketsDropCount"
  comparison_operator = "GreaterThanOrEqualToThreshold"
  threshold           = 1
  evaluation_periods  = 1
  period              = 60
  statistic           = "Sum"
  dimensions = {
    NatGatewayId = each.value
  }
  alarm_actions = [var.sns_topic_arn]
}
```

NAT Gatewayのアラームでは `for_each = var.nat_gw_ids` を使用しています。devは1台・prodはAZごとに配置という構成差があるため、NAT GatewayのID一覧をmapで受け取り、台数に関わらず同じコードで対応できるようにしています。

**ErrorPortAllocation**は、NAT Gatewayが送信元ポートを確保できなかった回数を表します。1件でも発生した時点でアプリケーションへの影響が出るため、閾値を1、発報間隔も1分と短く設定しています。発生する場合はPrivate SubnetのEC2から大量の外部接続が発生している可能性があります。

**PacketsDropCount**は、NAT Gatewayがドロップしたパケット数を表します。ポート不足と同様に1件から検知します。

## 5. logsモジュール

ログ収集は3種類を対象としています。

- EC2のシステムログ（`/var/log/messages`）
- Apacheのアクセスログ・エラーログ（`/var/log/httpd/`）
- VPCフローログ

### 5-1. CloudWatch Log Groups

```terraform
# EC2ログ収集用のLog Group
resource "aws_cloudwatch_log_group" "messages" {
  name              = "/${var.env}/ec2/messages"
  retention_in_days = var.log_retention_days
  tags = {
    Environment = var.env
  }
}

resource "aws_cloudwatch_log_group" "httpd_access" {
  name              = "/${var.env}/ec2/httpd/access"
  retention_in_days = var.log_retention_days
  tags = {
    Environment = var.env
  }
}

resource "aws_cloudwatch_log_group" "httpd_error" {
  name              = "/${var.env}/ec2/httpd/error"
  retention_in_days = var.log_retention_days
  tags = {
    Environment = var.env
  }
}

# VPCフローログ用のLog Group（条件付き作成）
resource "aws_cloudwatch_log_group" "vpc_flow" {
  for_each = var.enable_vpc_flow_logs ? { enabled = true } : {}

  name              = "/${var.env}/vpc/flowlogs"
  retention_in_days = var.log_retention_days
  tags = {
    Environment = var.env
  }
}
```

Log Groupの命名は `/{env}/対象/ログ種別` の規則で統一しています。`retention_in_days` はvariablesで受け取り、環境ごとに保持期間を変えられる設計です。devでは短め（例：7日）、prodでは長め（例：30日）に設定することでコストを調整できます。

VPCフローログ用のLog Groupは `for_each` を使った条件付き作成にしています。`enable_vpc_flow_logs = false` の場合、このリソースは作成されません。

### 5-2. VPCフローログとIAMロール

VPCフローログをCloudWatch Logsに書き込むには、専用のIAMロールが必要です。

```terraform
# フローログ書き込み用IAMロール
resource "aws_iam_role" "flowlogs" {
  for_each = var.enable_vpc_flow_logs ? { enabled = true } : {}

  name = "${var.env}-flowlogs-role"

  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect    = "Allow"
      Principal = { Service = "vpc-flow-logs.amazonaws.com" }
      Action    = "sts:AssumeRole"
    }]
  })
}

resource "aws_iam_role_policy" "flowlogs" {
  for_each = var.enable_vpc_flow_logs ? { enabled = true } : {}
  role     = aws_iam_role.flowlogs["enabled"].id

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect = "Allow"
      Action = [
        "logs:CreateLogStream",
        "logs:PutLogEvents",
        "logs:DescribeLogGroups",
        "logs:DescribeLogStreams"
      ]
      Resource = [
        aws_cloudwatch_log_group.vpc_flow["enabled"].arn,
        "${aws_cloudwatch_log_group.vpc_flow["enabled"].arn}:*"
      ]
    }]
  })
}

# VPCフローログ有効化
resource "aws_flow_log" "this" {
  for_each = var.enable_vpc_flow_logs ? { enabled = true } : {}

  vpc_id               = var.vpc_id
  traffic_type         = "ALL"
  log_destination_type = "cloud-watch-logs"
  log_destination      = aws_cloudwatch_log_group.vpc_flow["enabled"].arn
  iam_role_arn         = aws_iam_role.flowlogs["enabled"].arn
}
```

IAMロールのAssumeRolePolicyでは `vpc-flow-logs.amazonaws.com` を信頼するServiceとして設定しています。インラインポリシーで付与する権限は最小限に絞り、対象のLog GroupのARNのみをResourceに指定しています。

`traffic_type = "ALL"` としているため、ACCEPT・REJECT両方のトラフィックが記録されます。不正アクセスの試みや予期せぬ通信パターンの事後追跡に活用できます。

なお、EC2アプリケーションログ（`/var/log/messages` や `/var/log/httpd/`）をCloudWatch Logsへ転送するには、EC2上にCloudWatch Agentのインストールと設定が必要です。この記事では **Log Groupの作成までをTerraformで管理** し、CloudWatch AgentのEC2への展開はSSM Parameter StoreとUser Dataを組み合わせた方法で別途対応します（後続記事で扱う予定です）。

## 6. root/main.tf への組み込み

各モジュールをルートから呼び出す構成は以下の通りです。

```terraform
# SNS
module "sns" {
  source     = "../../modules/sns"
  send_email = var.send_email
  env        = var.env
}

# CloudWatch Logs
module "logs" {
  source               = "../../modules/logs"
  env                  = var.env
  enable_vpc_flow_logs = var.enable_vpc_flow_logs
  log_retention_days   = var.log_retention_days
  vpc_id               = module.vpc.vpc_id
}

# CloudWatch Alarms
module "alarms" {
  source        = "../../modules/alarm"
  env           = var.env
  sns_topic_arn = module.sns.sns_topic_arn

  asg_name               = module.ec2.asg_name
  alb_arn                = module.alb.alb_arn
  alb_tg_arn             = module.alb.alb_tg_arns
  db_instance_id         = module.rds.db_id
  db_connections_threshold = var.db_connections_threshold
  nat_gw_ids = {
    "az-a" = module.vpc.nat_gw_ids[0]
  }
}
```

既存の `vpc` / `ec2` / `alb` / `rds` モジュールの変更はほぼゼロです。各モジュールが公開済みのoutput（`asg_name` / `alb_arn` / `alb_tg_arns` / `db_id`）を新モジュールへ渡すだけで監視基盤を後付けできます。

これは前回記事で設計した「モジュール間はvariables / outputsでIF設計する」方針の効果です。依存方向を一方向に保っていたことで、既存モジュールへの影響なしに機能追加ができています。

## 7. 今回の実装でカバーできたこと・できなかったこと

### ✅ カバーできたこと

| 対象 | 実装内容 |
|---|---|
| EC2 | CPU使用率・死活監視（StatusCheckFailed）|
| ALB | Unhealthyホスト検知・5xxエラー・レスポンスタイム |
| RDS | CPU・接続数・ストレージ・メモリ残量 |
| NAT GW | ポート不足・パケットドロップ |
| VPCフローログ | CloudWatch Logsへの転送（IAMロール含む） |
| EC2ログ | Log Groupの作成（`/messages` / `/httpd/access` / `/httpd/error`）|
| 通知基盤 | SNS Topicによるメール通知 |

### ❌ 今回のスコープ外

- **CloudWatch AgentのEC2への展開**：Log GroupはTerraformで作成済みだが、EC2上へのAgent設定ファイルの配置・起動はSSM連携で別途対応予定
- **ALBアクセスログのS3出力**：別途S3バケットの作成が必要なため後続で対応予定
- **ログベースのアラーム**（Metric Filter）：アプリログのエラーパターンを検知するMetric Filterは、Agent展開後に対応予定
